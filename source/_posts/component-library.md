# 背景

上一次将所有项目的重复工具函数提取出来后, 本以为可以进行下一步, 但是有开发提出使用按需加载后, 打包的体积反而变大了

> 按需加载是另外一个开发做的

经过排查定位, 确定是我之前抽取公共函数导致的(～￣▽￣)～, 然后我又一番排查, 发现了问题所在

之前组件库也有一部分工具函数和业务项目重复了, 所以我顺手将组件库的工具函数放到了公共函数里

为了图方便, 所以

```JavaScript
// basicWidget/common.js
export * from '@/widgets/basic/common-functions'
```

我在这里直接把所有公共函数都给导出了, 之前没问题, 是因为还没有添加业务项目的公共函数, 这次添加后, 体积暴涨....

相当于所有独立打包的组件都将公共函数给打进去了

## 定位过程

初步分析是按需加载多次引入common-function导致, 但结果稍微有出入, 因为旧项目的common-function只有20KB, 很难导致700KB的变化


1.	旧组件库引入与不引入common-function, 单个组件大小变化不大, 按需加载的总体积非常大
2.	vms使用旧组件库, 使用按需加载后, tvtcloudbaiscwidget变得巨大
3.	vms 使用旧组件库排除common-function, 按需加载 变小(0.2M)
4.	devops 使用最新版本 按需加载后, 体积变大
使用旧组件库 排除common-function, 按需加载后 体积变小
5.	新组件库打包后总体体积变得巨大 (已确定, 是ali-oss的原因)

单个组件更是逆天, 以tvt-date-picker为例, 直接翻了七倍(201KB => 1.43MB)

深入分析后

目前怀疑三个方向综合因素导致变得巨大
1.	每个单独的组件都引入了common-function, 体积变大, 在新组件库非常明显
2.	按需引入的组件都包含运行时, 一旦使用的组件变多, 按需加载的优势反而会变成劣势(平均130kb, 只要超过了13个组件, 体积反而会变大, 全量引入是1.8MB)
3.	vms使用的旧组件库, 可能将全量和按需加载都引入了,不然不可能变化这么明显
4.	ali-oss和aws-sdk引入的三方件非常大

最后结论, 正是上述三个方面同时作用导致的, 第三点的原因和第二点一样, 因为vms按需加载了特别多的组件(25个), 所以增加很明显


## 思路

导出问题很好解决, 在common.js里面导出组件库需要使用的函数即可

但是这只能解决表面的问题, 按需加载后, 打包体积变大还隐藏着一个更深层次的问题, 那就是所有组件都有相当一部分重复的代码:

1. 相同的运行时代码
2. 重复的第三方依赖
3. 重复的内部依赖

> 特别是其它开发搞的按需加载是umd导出

@图7

可以看到tvt-date-picker除了自身代码外, 还有其它代码

### 按需导入

关于按需导入, 这是一个老生常谈的问题, 社区开源的组件库(element ui, ant-design)都提供了较为成熟的方案

方案一:
组件库内部管理重复的代码, 将这些重复的模块标记为external, 单个组件打包时不会将它们打包进去, 最小化导出的组件

再配合webpack的按需导入插件, 可以做到非常方便的使用

@图8

引入自己内部模块

@图9

将内部模块以及第三方模块标记为external

方案二:

es module导出, 让业务项目自己进行转换代码, 更加灵活

@图10

由于目前业务项目打包的速度已经够慢了(40多秒), 我不想增加时间, 所以只能从方案一入手

## 解决方案

1. 将一些重量级的第三方依赖声明为external,比如vue, element-ui, ali-oss, aws-sdk
2. 先删除组件库里从未被使用的工具函数以及模块, 降低内部依赖的体积
3. 内部共享一些常见的内置模块, 避免重复打包, 比如 @ui/button, @ui/locale,  @js/date-tools

具体步骤就不说了, 比较简单, 就是写一些external判定, 可以参考element-ui

> 之所以没有完全整改共享内置模块, 那是因为改动比较大
